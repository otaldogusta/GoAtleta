# CONSULTORIA - Checklist do piloto real

Use este checklist para testar o fluxo local com uma aluna real antes de desenhar o schema Supabase da CONSULTORIA 2.

## Objetivo do piloto

Validar o contrato minimo:

`perfil -> treino -> execucao -> PSE/dor -> feedback -> revisao`

Nao validar escala, automacao, notificacoes ou relatorio formal nesta etapa.

## Preparacao

- Selecionar uma aluna real.
- Preencher o perfil de treino.
- Conferir objetivo, ambiente, materiais, restricoes, lesoes, frequencia e duracao.
- Criar pelo menos um treino da semana.
- Publicar o treino.
- Confirmar que a aluna sabe onde acessar `Treino online`.

## Checklist da aluna

- Conseguiu acessar o treino?
- Entendeu o objetivo do treino?
- Entendeu a ficha `Atividade | Series | Repet. | Interv. | Obs.`?
- Conseguiu iniciar e concluir o treino?
- Entendeu a escala de PSE?
- Entendeu a escala de dor?
- Enviou uma observacao util?
- Sentiu falta de video ou demonstracao?
- Sentiu falta de alguma orientacao de seguranca?

## Checklist do professor

- Conseguiu preencher o perfil sem demora excessiva?
- A prescricao ficou clara no formato de ficha?
- Conseguiu publicar o treino sem duvida?
- O feedback recebido ajudou a tomar decisao?
- A PSE foi suficiente para ajustar o treino?
- O nivel de dor foi suficiente para decidir cuidado/reducao?
- A revisao ficou clara no fluxo?
- Conseguiu decidir a proxima semana com base no retorno?

## Campos para observar

Nao implementar ainda. Apenas anotar se aparecerem no uso real.

- Nivel de experiencia da aluna.
- Horario preferido.
- Equipamento real disponivel.
- Espaco disponivel em casa.
- Exercicio que a aluna nao consegue fazer.
- Exercicio que a aluna nao gosta.
- Carga usada.
- Dificuldade por exercicio.
- Video de execucao enviado pela aluna.
- Video demonstrativo do exercicio.
- Dor por exercicio, nao apenas no treino inteiro.

## Criterios para avancar para CONSULTORIA 2

- A aluna concluiu pelo menos uma semana de uso.
- O professor conseguiu revisar os feedbacks.
- O contrato de dados atual foi suficiente ou as lacunas foram listadas.
- Campos novos foram priorizados com base no uso real.
- Riscos de permissao aluno/professor foram revisados.
- Decisao tomada sobre manter ou alterar o modelo antes da migration.

## Decisoes para registrar

- Quais campos entram no Supabase agora?
- Quais campos ficam para historico/progresso?
- Quais campos ficam para midia?
- Quais notificacoes sao realmente necessarias?
- O que precisa ser obrigatorio?
- O que deve continuar opcional?

## Nao fazer durante o piloto

- Nao prometer diagnostico.
- Nao prometer resultado.
- Nao usar como substituto de avaliacao presencial quando ela for necessaria.
- Nao criar Supabase antes de validar os campos reais.
- Nao adicionar notificacoes antes de ter evento canonico.
